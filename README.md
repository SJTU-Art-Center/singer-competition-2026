# 校园歌手比赛可视化管理系统 (Singer Competition Visualization)

本项目是一个为校园歌手大赛专门定制的计分与状态展示系统。包含后台系统与前台大屏全屏特效展示，前后端通过 WebSocket 实时同步比赛各选手状态。

## 赛制规则

本项目严格按照以下赛事规则开发并自动化运行推演：

- **第一轮：30 进 18 排位突围赛**
  - 30 位选手赛前抽签，每 5 人一组演唱。
  - **评分体系**：总评分 = 专业评委分 (75%) + 大众评审分 (25%)。
  - **排名流转等级**：
    - **第 1 - 2 名**：获得“大魔王”称号（具备优先晋级保送权）。
    - **第 3 - 10 名**：进入守擂区，成为“擂主”。
    - **第 11 - 18 名**：进入挑战区，成为“挑战者”。
    - **第 19 - 30 名**：直接淘汰收场。

- **第二轮 环节一：1V1 攻守擂台战**
  - 第 11 至 18 名的挑战者，依排名顺序从守擂区挑选对手进行 1v1 PK。
  - **胜负判定界限**：
    - 若擂主分数 > 挑战者分数：擂主直接晋级十强，挑战者直接遭到淘汰。
    - 若挑战者分数 > 擂主分数：挑战者不得直接晋级，而是与守擂失败的擂主并列双双掉入“待定池”。

- **第二轮 环节二：大魔王荣耀返场**
  - 两名“大魔王”依次返场表演并接受打分。
  - **严格红线防守机制**：大魔王第二轮的得分必须**严格大于**环节一中 16 位选手的首轮成绩平均分，才能直接空降十强（等于平均分均视同闯关失败）。如果不达标，将被剥夺保送资格被抛入“待定池”。

- **终极十强补位**
  - 统计已拥有的直通席位名额后，剩余的空缺将全部从“待定池”中抽调填补。
  - **同分对冲机制**：池内所有选手按第二局分数从高到低补位，若出现由于比分平局面临排位争锋，系统将自动回溯参考选手在**第一轮的原始成绩**做出最终判定抉择。

## 技术架构

- **前端 (Client)**: React.js, Vite, TailwindCSS (组件式极速样式开发), React Router, Socket.io-client。
- **后端 (Server)**: Node.js, Express, Socket.io (WebSocket 长链接无延迟推流)。
- **数据存储**: 基于文件存储的纯本地 JSON 文件 (无外部冗余数据库依赖)。

## 项目启动方法

### 后端 (服务端与状态中心)
1. 进入 `server` 目录：
   ```bash
   cd server
   ```
2. 安装依赖并启动：
   ```bash
   npm install
   node index.js
   ```
   > 默认后端会在 `3001` 端口建立服务，自动生成存储数据的文件 `data.json`。

### 前端 (管理台与大屏)
1. 进入 `client` 目录
   ```bash
   cd client
   ```
2. 安装依赖并启动前端页面：
   ```bash
   npm install
   npm run dev
   ```
   > 如果是在同一台物理机上启动的本地环境，直接访问控制台提示的地址 (通常为 `http://localhost:5173` )

## 模块介绍

- **后台管理系统**：入口位于 `/admin`。主要供执行导演或后台控制统筹员使用操作，在这里支持：
   - 切换比赛各个论述。
   - 输入专业评委分和大众评分。
   - 配置 1v1 的对战顺序与打分。
   - 进行自动化的大数据模拟补位算例执行。
- **舞台大屏**：入口位于 `/screen`。主要在副屏、场馆主显示屏投放。该界面隐去了所有的调参按钮和交互反馈，依靠 WebSocket 仅呈现动效并播放在后台的操作实况。
